# automation/

The SDET-owned automated test project. **All test code generated by the `test-auto` skill lands here** (unless `$TEST_PROJECT_PATH` overrides the location).

This folder is intentionally empty in the template — initialize it with whatever framework your team uses.

## Initialize a framework

Run one of these from inside `automation/`:

| Stack | Init command |
|---|---|
| Playwright (TypeScript) | `npm init playwright@latest` |
| Playwright (Python)     | `pip install playwright pytest-playwright && playwright install` |
| Cypress                 | `npm init -y && npm i -D cypress && npx cypress open` |
| xUnit (.NET)            | `dotnet new xunit -n Automation` |
| NUnit + SpecFlow (.NET) | `dotnet new nunit -n Automation && dotnet add package SpecFlow.NUnit` |
| pytest (Python)         | `python -m venv .venv && pip install pytest pytest-bdd requests` |
| RestAssured (Java)      | `mvn archetype:generate -DgroupId=qa.tests -DartifactId=automation -DarchetypeArtifactId=maven-archetype-quickstart` |

The `test-automator` agent reads existing tests here before generating new ones — so once you have **one** test in your chosen style, every subsequent generated test will follow that style.

## Recommended structure

The agents are framework-agnostic and will mirror whatever they see, but a typical layout works well:

```
automation/
├── tests/         # the test files themselves
├── pages/         # Page Object Model classes (UI)
├── api/           # API client helpers
├── fixtures/      # shared setup, test data factories
├── support/       # custom commands, utilities
└── <config files for the framework>
```

## Overriding the location

If your team already has an automated test project elsewhere (in the product repo, in a separate solution, etc.), point the skills at it:

- Set `TEST_PROJECT_PATH=<absolute path>` in `.env`, or
- Pass the path as an argument to `/test-auto`.

When `$TEST_PROJECT_PATH` is set, the skills ignore this `automation/` folder entirely.

## What NOT to put here

- The product under test — that's separate, set `PROJECT_UNDER_TEST` to its path.
- Manual test cases — those go in `test-cases/`.
- One-off scripts, scratch files — keep this folder clean and run-ready.
