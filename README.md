# QA Standard — Claude Code Template for SDET Teams

A reusable Claude Code template for QA / SDET workflows: feature testing, test-case authoring, test automation, code investigation, and security scanning. Drop this template into a new project, configure the integrations you need, and start running the workflows.

## What's inside

```
.claude/
├── settings.json            # project permissions + env defaults
├── project-facts.md         # generated by /bind-project — canonical product facts
├── skills/                  # invokable workflows (`/skill-name` or via Skill tool)
│   ├── bind-project/        # adapt this template to a specific product (RUN FIRST)
│   ├── auto-setup/          # scaffold a thoughtful Playwright + TS project from a blueprint
│   ├── auto-enhance/        # audit an existing Playwright project + plan architecture fixes
│   ├── test-feature/        # end-to-end PR/story sign-off workflow
│   ├── test-docs/           # build manual test cases from a user story
│   ├── test-auto/           # convert manual test cases into automated tests
│   ├── test-find/           # multi-agent code investigation for a category
│   └── security-scan/       # generate an HTML security report
└── agents/                  # specialist subagents the skills orchestrate
    ├── pr-reviewer.md
    ├── qa-reviewer.md
    ├── ui-tester.md
    ├── api-tester.md
    ├── ba-analyst.md
    ├── test-case-writer.md
    ├── test-automator.md
    ├── playwright-architect.md  # used by /auto-setup + /auto-enhance to materialize a blueprint
    ├── automation-auditor.md    # used by /auto-enhance to audit one architectural lens
    ├── security-auditor.md
    ├── code-investigator.md
    └── project-mapper.md    # used by /bind-project to map a real codebase

automation/        # the SDET-owned test project — `test-auto` writes automated tests here
                   # (override with $TEST_PROJECT_PATH if your team's tests live elsewhere)
questions/         # `test-find` writes open questions here (markdown per topic)
security/          # `security-scan` writes the HTML report here
test-cases/        # `test-docs` writes manual test cases here (override with $TEST_CASES_DIR)
```

Initialize a framework inside `automation/` once (Playwright, xUnit, pytest, etc.) — see [automation/README.md](automation/README.md). The `test-automator` agent reads existing tests and mirrors their style, so after the first test, subsequent generated tests stay consistent.

## Skills (quick reference)

| Skill | Purpose | Inputs | Outputs |
|---|---|---|---|
| `bind-project` | **Run first.** Adapt this template to a specific real product. Spawns recon agents that map the stack, app surface, test infrastructure, and integrations. | Absolute path to product source (and optional portal tokens via env) | `.claude/project-facts.md`, updated `CLAUDE.md` (project-specific block) and `settings.json` env defaults |
| `auto-setup` | Bootstrap a Playwright + TS test project with thoughtful architecture (multi-project config, base POM, typed API client, validated multi-env loader, fixtures, factories, tags, CI stub). | Target path (default `automation/`), reads `.claude/project-facts.md` for product context | Full scaffolded project at target path, ready to add real tests against |
| `auto-enhance` | Audit an existing Playwright + TS project across six architectural lenses (config, POM/locators, fixtures/auth, env, API/E2E separation, test quality) and produce a prioritized remediation plan. | Target path of an existing Playwright project | `audit-<ts>.md` inside the test project; optional `--apply` mode delegates fixes to `playwright-architect` |
| `test-feature` | Decide if QA can close a user story / PR | PR URL or task description (and optional portal+PAT) | Verdict (close / don't close), bug list, coverage & automation recommendations |
| `test-docs` | Build manual test cases for a user story | User story (text or ID), source project path, test project path | Markdown test cases in `test-cases/` |
| `test-auto` | Convert a manual test case into an automated test | Manual test case file (or text) | Automated test file written into `automation/` (or `$TEST_PROJECT_PATH`), matching the existing framework |
| `test-find` | Multi-agent code review for a category (security, performance, a11y, …) | Category name | Findings + questions for the BA written to `questions/<topic>.md` |
| `security-scan` | Full-codebase security audit | (none) | `security/security-report-<timestamp>.html` |

## Getting started

1. **Copy this template** into the root of the project you want to QA, OR keep it as a standalone workspace that references the project under test.
2. **Bind to your product** — run `/bind-project <absolute-path-to-product>` once. This is the adapter: it explores the real codebase via parallel recon agents, then writes `.claude/project-facts.md`, updates the project-specific section of `CLAUDE.md`, and fills the non-secret env defaults in `.claude/settings.json` (stack, base URLs, test project path). The chat summary lists which secrets (`TEST_USER`, `TEST_PASSWORD`, `*_PAT`, `*_TOKEN`) you still need to add to your local `.env`. Re-run any time the product's shape changes.
3. **Configure secret integrations** in `.env` (the only file `bind-project` will not write for you):
   - Azure DevOps PAT → `AZDO_PAT` (and confirm the org/project `bind-project` detected)
   - Jira token → `JIRA_TOKEN` (and `JIRA_BASE_URL`, `JIRA_PROJECT`)
   - GitHub token → `GH_TOKEN`
   - Test user credentials → `TEST_USER`, `TEST_PASSWORD`
   - Playwright / Chrome DevTools MCP for UI testing
   - REST / OpenAPI MCP for API testing
4. **Invoke a skill** by typing its slash command (e.g. `/test-feature PR 12345`) or asking Claude to run it. After binding, the skills will reference the real selectors, endpoints, and conventions of your product.

## How the skills delegate to agents

Each skill is a coordinator — it does not do the work itself. It spawns specialist subagents (see `.claude/agents/`) in parallel where possible, collects their structured findings, and produces the final artifact. This keeps the main context window clean and lets each specialist focus narrowly.

See [CLAUDE.md](CLAUDE.md) for the deeper architectural notes and house rules.
